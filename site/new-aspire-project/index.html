The empty template is the easy part.

A new Aspire app gives you a blank API and a frontend. The layering, the CQRS slices, the migrator, the database wiring: that is the day you lose to plumbing. new-aspire-project hands you that day finished, named for your project, and already building.

A real CQRS slice, per feature

Every feature gets its own vertical slice in the Application layer: commands with FluentValidation validators, queries, DTOs, a static mapper, and a typed exception. Handlers use C# primary constructors for dependency injection. The API layer stays thin, delegating to MediatR.

Four databases, one command

PostgreSQL, SQL Server, MySQL or SQLite. The skill rewrites the packages, the AppHost resource and the DbContext registration to match, and respects Central Package Management.

A migrator that runs first

A dedicated DbMigrator console app applies EF Core migrations and seeds sample data on startup. The API waits for it, so it never boots against an empty schema.

Scaffold it, then run it.

Give the skill a name and a database. It copies the template to your repo root, renames everything, configures the database, builds, and generates the InitialCreate migration. Then one command brings the whole solution up.

  • Renames every project, namespace, reference and solution entry.
  • Runs dotnet build to catch problems before you do.
  • ServiceDefaults wires OpenTelemetry, health checks and resilience.

Targets .NET 10 SDK. Redis caching stays configured for every database choice.
