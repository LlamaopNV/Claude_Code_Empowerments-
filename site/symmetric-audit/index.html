The idea

Every change belongs to a family.

Most domain processes are not a single file. They are a family of surfaces: create plus edit plus view, list plus detail plus mutation, insert plus update, loader plus action plus form. A change to one member is rarely complete in isolation.

Touch one cell below and the rest of the family is implicated. The question is never "did this file change correctly", it is "did its siblings change too, or did one fall silent".

One write touches the whole family. The audit makes that family explicit.

How it finds the siblings.

No memory, no guessing. It pulls the entity name out of the change set, then greps that entity's full CRUD lifecycle across the repo. Five buckets, run for each entity, then unioned.

entity-driven CRUD grep entity: order
$grep -rn "order" src/ | grep -i \\
  "find\|query\|select\|get\|read"
A orders.list.query read
A order.detail.query read
 
$grep -rn "order" src/ | grep -i \\
  "create\|insert\|update\|save"
B order.write write
 
$find src/ -iname "*order*"
E order-edit.form component
E order-detail.view component
~ order-export.csv B + E intersection
ADatabase reads

Every query, find, select, get, or fetch of the entity. The read side that must surface a new field.

BDatabase writes

Every create, insert, update, upsert, save, or remove. The write twins that should accept the same input.

CAPI and RPC calls

Every place the frontend calls this entity's endpoints: routers, procedures, mutations, queries.

DRoute and page files

The route or page directory for the entity: the surfaces a user actually lands on.

EComponents named after the entity

Forms, lists, badges, detail views: anything carrying the entity name. Drop tests and build output, and the union is your sibling set.

A sibling flagged by multiple buckets is a high-priority intersection. Those files are the most likely to hide the asymmetric bug. Audit them first.

More than about 30 siblings means the change set is too broad for one audit. That is the signal to split the work into smaller pull requests.

The divergence classes it catches.

The gate to a "diverged" verdict is strict. The audit reads each sibling and compares it against the change, looking for one of these specific gaps. The last one is not a bug at all.

forgotten edit form
 // create accepts the new field
+create.form: { name, email, phoneNumber }
!edit.form: { name, email } // phoneNumber lost

Forgotten edit form. A field added to create never reached the edit twin. Users can set it once, then never change it.

unguarded route
+/reports guard: admin
!/reports/export public

Unguarded sibling route. A guard landed on one route. Its sibling in the same family is still public.

half-handled enum
+badge: case REFUNDED
!filter: no REFUNDED, throws

Half-handled enum. A new status is handled in one switch and crashes another that never learned the value.

projection gap
+write("orders", { ..., archivedAt }) // persisted
!read.select: [id, total, status] // archivedAt absent

Projection gap. The field is written to the row but missing from the read projection, so the UI can never see it.

mutation contract drift
+updateOrder(id, patch, reason) // reason now required
!caller: updateOrder(id, patch) // missing reason

Mutation contract drift. A mutation gained a required argument. A caller in the family still passes the old shape.

deliberate omission
+create.input: { email, password }
~edit.input: no password, by design

Deliberate omission. Genuinely asymmetric, not a bug. The audit writes down the reason so the next reader can tell intent from oversight.

Worked example

The summary is the work product.

One real change: a developer adds archivedAt to orders, editing only the write path and the create form. Two files in the diff. Then the audit walks the whole order family and writes a verdict for every sibling, because silence on a sibling is how the gap slips through.

Before the audit

2 files changed

3 siblings silently diverged. It compiles, review passes, it ships green. The gap is invisible until production reads a column that was never written to the projection.

After the audit

7 siblings considered

Every sibling carries a written verdict. The 3 diverged are fixed in this same pass, the 1 intentional asymmetry is documented, and the in-sync ones are named so nothing is assumed.

Run it as your pre-commit gate.

It is entity-driven and stack-agnostic. Point it at a change set, it greps the full CRUD lifecycle, renders a verdict per sibling, and fixes the diverged ones before the commit lands.

  • Walks create, read, update, delete and render surfaces.
  • Writes a verdict for every sibling, including the in-sync ones.
  • Step one of the iterative-review-fix loop.

Reach for it

At a coherent stopping point, before commit, after touching a schema, a shared validator, or a shared form section.

Skip it

Mid-implementation while work is incoherent, on a one-file typo fix, or when the user explicitly scoped the change to one surface.
