# new-aspire-project

A skill to bootstrap a new .NET Aspire project from the AspireApp1 template.

## Description

This skill helps you create a new .NET Aspire project by:
1. Asking for your project name and preferred database
2. Copying the AspireApp1 template folder
3. Renaming all projects, namespaces, and references to match your project name
4. Configuring your chosen database (PostgreSQL, SQL Server, MySQL, or SQLite)
5. Building the project to ensure everything works
6. Generating EF Core migrations automatically

## Usage

```
/new-aspire-project
```

Or with command line arguments:
```
/new-aspire-project MyProjectName SqlServer
```

## What This Skill Does

### Step 1: Gather Requirements
- Prompts for the new project name (e.g., "MyAwesomeApp")
- Asks which database to use:
  - **PostgreSQL** (default) - Using Aspire.Npgsql.EntityFrameworkCore.PostgreSQL
  - **SQL Server** - Using Aspire.Microsoft.EntityFrameworkCore.SqlServer
  - **MySQL** - Using MySql.EntityFrameworkCore (Oracle's official provider)
  - **SQLite** - Using Microsoft.EntityFrameworkCore.Sqlite

### Step 2: Copy and Rename
- Copies the AspireApp1 template folder to a new directory at the **git repo root**
- Renames all .csproj files
- Updates the solution file
- Replaces all namespace references from AspireApp1 to your project name
- Updates using statements, class references, and project references
- Updates the AppHost.cs file to reference updated projects

### Step 3: Database Configuration
- Updates the Infrastructure.csproj with the correct database package references
- Updates AppHost.cs with the appropriate database resource
- Ensures the DbMigrator references the correct database
- **Creates ApplicationDbContextFactory.cs** - Required for EF Core migrations

### Step 4: Build and Verify
- Runs `dotnet build` on the solution
- Reports any errors if the build fails
- Confirms successful project creation

### Step 5: Generate Migrations
- Automatically runs: `dotnet ef migrations add InitialCreate` in the Infrastructure project
- This creates the database schema migration files
- Without migrations, the application will fail with "no such table" errors
- The ApplicationDbContextFactory.cs created in Step 3 enables this command to work

## Project Structure

The created project will have this structure:
```
YourProjectName/
├── YourProjectName.slnx
├── Directory.Packages.props
├── Orchestration/
│   ├── YourProjectName.AppHost/
│   └── YourProjectName.ServiceDefaults/
├── Source/
│   ├── YourProjectName.Api/
│   ├── YourProjectName.Application/
│   ├── YourProjectName.Domain/
│   ├── YourProjectName.Infrastructure/
│   ├── YourProjectName.DbMigrator/
│   └── yourprojectname-client/        (Next.js frontend, lowercase)
└── Tests/
    └── YourProjectName.Application.Tests/
```

## Requirements

- .NET 10.0 SDK or later
- AspireApp1 template folder (included in the skill directory)
- Write permissions in the current directory

## Important: Central Package Management

The AspireApp1 template uses **Central Package Management (CPM)**, which means:
- Package versions are defined centrally in `Directory.Packages.props`
- Individual `.csproj` files reference packages WITHOUT version numbers
- When changing databases, BOTH the `.csproj` files AND `Directory.Packages.props` must be updated
- Failure to update `Directory.Packages.props` will result in NU1010 errors during restore

## Examples

Create a project named "ECommerce" using SQL Server:
```
/new-aspire-project ECommerce SqlServer
```

Create a project named "BlogEngine" using PostgreSQL:
```
/new-aspire-project BlogEngine PostgreSQL
```

## Notes

- Deterministic file operations (copy, rename, replace, database config) are handled by `bootstrap.py`
- The skill invokes the script, then handles build, migrations, and error recovery
- **Central Package Management**: The template uses CPM - the script must update Directory.Packages.props for any package changes
- **Migrations are automatically generated**:
  - Existing PostgreSQL migrations are removed during setup
  - New migrations for the selected database are automatically generated by the skill
  - ApplicationDbContextFactory.cs is created to enable migration generation
  - Migrations are applied by DbMigrator when the application starts
- The skill performs a full build to catch any issues early
- Redis caching remains configured regardless of database choice
- **MySQL-specific**: Uses Oracle's `MySql.EntityFrameworkCore` (not Pomelo). Pomelo 9.0 is incompatible with EF Core 10. There is no Aspire EF Core integration for Oracle's provider, so MySQL uses manual DbContext registration with `UseMySQL()` and reads the connection string from Aspire configuration — same pattern as SQLite.
- **SQLite-specific**: Uses absolute paths to a shared database file in the solution root to ensure all projects (API, DbMigrator) access the same database file. Without this, each project would create its own separate database.

**Key Files for Database Configuration:**
- `Directory.Packages.props` - Package versions (CPM)
- `Source/{ProjectName}.Infrastructure/{ProjectName}.Infrastructure.csproj` - Infrastructure packages
- `Orchestration/{ProjectName}.AppHost/{ProjectName}.AppHost.csproj` - Hosting packages
- `Source/{ProjectName}.DbMigrator/{ProjectName}.DbMigrator.csproj` - Migrator packages
- `Orchestration/{ProjectName}.AppHost/AppHost.cs` - Database resource configuration
- `Source/{ProjectName}.Api/Program.cs` - AddInfrastructure call (signature changes for MySQL/SQLite)
- `Source/{ProjectName}.Infrastructure/DependencyInjection.cs` - DbContext registration
- `Source/{ProjectName}.DbMigrator/Program.cs` - Migrator DbContext registration
- `Source/{ProjectName}.Infrastructure/Data/ApplicationDbContextFactory.cs` - Design-time factory (MUST be created for EF migrations)
- `Source/{projectname}-client/generate-api.ps1` - API client generation script (has path references to rename)
- `{RepoRoot}/.aspire/settings.json` - Aspire settings (appHostPath must point to new project)

## Implementation Instructions

When this skill is invoked, follow these steps:

### Phase 1: Collect Information

1. Parse command line arguments if provided (projectName and databaseType)
2. If not provided via arguments, use AskUserQuestion to gather:
   - **Project Name**: The name for the new project (validates it's a valid C# identifier)
   - **Database Type**: Choice between PostgreSQL, SQL Server, MySQL, or SQLite
3. Map the database choice to the script's `--db` value:
   - PostgreSQL → `postgresql`
   - SQL Server → `sqlserver`
   - MySQL → `mysql`
   - SQLite → `sqlite`

### Phase 2: Run the Bootstrap Script

All deterministic file operations (copy, rename, replace, database config, context factory, aspire settings) are handled by `bootstrap.py`, located next to this skill file.

**IMPORTANT — Shell environment:** The shell is **bash** (Git Bash on Windows), NOT PowerShell.

1. Determine the **git repository root** and the **plugin directory** (the directory containing this instruction file, `bootstrap.py`, and the `AspireApp1/` template):
   ```bash
   REPO_ROOT=$(git rev-parse --show-toplevel)
   ```

   **Finding the plugin directory:** The `bootstrap.py` script and `AspireApp1/` template folder are located in the same directory as this instruction file. Use the known path of this file to resolve the plugin directory.

2. Run the bootstrap script:
   ```bash
   python "<plugin-dir>/bootstrap.py" \
     --name {ProjectName} \
     --db {db_type} \
     --template "<plugin-dir>/AspireApp1" \
     --output "$REPO_ROOT"
   ```

   Where `<plugin-dir>` is the absolute path to the directory containing this instruction file.

3. If the script exits with code 0, proceed to Phase 3
4. If the script exits with non-zero, report the error output to the user and stop

The script handles:
- Copying template (excluding bin, obj, node_modules, .vs, .next, Migrations)
- Renaming all directories and files (including lowercase `aspireapp1-client`, `.esproj`, `.http`, `.json`)
- Replacing all `AspireApp1`/`aspireapp1` references in file contents (`.cs`, `.csproj`, `.slnx`, `.json`, `.props`, `.ps1`, `.http`, `.md`, `.ts`, `.tsx`, `.js`, `.esproj`)
- Verifying no template references remain
- Database-specific package and code changes
- Creating `ApplicationDbContextFactory.cs`
- Updating `.aspire/settings.json`

### Phase 3: Build and Verify

1. Navigate to the new project directory:
   ```bash
   cd "$REPO_ROOT/{ProjectName}"
   ```
2. Run `dotnet restore && dotnet build --no-restore`
3. If build fails, report errors and troubleshoot
4. If build succeeds, continue to Phase 4

### Phase 4: Generate Migrations

1. **Generate EF Core migrations** (CRITICAL - do not skip):
   ```bash
   cd "$REPO_ROOT/{ProjectName}/Source/{ProjectName}.Infrastructure"
   dotnet ef migrations add InitialCreate
   ```
   - Verify the Migrations directory was created with migration files
   - If this fails, report the error to the user

2. **Final build**: Run `dotnet build` one last time from the solution root to confirm everything compiles cleanly with the new migrations

### Phase 5: Provide Next Steps

1. Inform the user the project is ready to run
2. Provide the command to run the application:
   ```bash
   cd {ProjectName}
   dotnet run --project Orchestration/{ProjectName}.AppHost/{ProjectName}.AppHost.csproj
   ```
3. Explain what will happen when they run it:
   - AppHost orchestrator starts
   - DbMigrator applies migrations and seeds sample data
   - API starts with Scalar documentation
   - Redis cache starts
4. Provide the URL for API documentation (e.g., http://localhost:5000/scalar/v1)

### Error Handling

- If `bootstrap.py` fails, show the full output and stop
- If project name is invalid, the script will report the error — ask for a new name
- If build fails, show the full error output and suggest potential fixes

**Common errors to watch for:**

**CPM-related:**
- **NU1010**: PackageReference without PackageVersion
- **NU1605**: Package downgrade detected (e.g., SQL Server needs 10.0.1, not 10.0.0)

**Migration-related:**
- **"Unable to create DbContext"** — ApplicationDbContextFactory.cs is missing or has incorrect syntax
- **Migration generation fails** — Check that EF Core tools are installed (`dotnet tool install --global dotnet-ef`)
